Object orientation

The goal of object oriented programming is to be able to add new features without altering existing code

Many books focus on the technical aspects of object oriented programming. They will explain how to define a class and how the mechanics of inheritence work. But they often fail to explain the goal of these techniques. Because of this, the true merics of object orientation are often lost. The result is an increasing the cost of a software project.

The ultimate goal of object orientation is to be able to add new features to a program without changing the already existing code. Very concretely this means that we can add new functionality simply by adding some subclasses of existing classes and optionally adding some new methods to existing classes, but we cannot alter existing methods. The invers of object orientation is: in order to support one additional feature, all existing classes would have to be modified. It is clear that if this would be true, adding new feature will become increasingly difficult.

The invers of object orientation is that, in order to add one feature, every exiting class needs to be changed.

Before we dive into the deeper parts on how to accomplice this goal, let give a quite recap of the basics of object oriented programming.

Separating goal from implementation

Does your mother know how the remote control of her television set works? Probabily not (except of she likes science and is a super cool mom). Yet, she is able to turn of the television. Wonderfull is it not! This means that when technology evolves and a new type of remote controll enters the markets, based on quantum emissions, you mother will still be able to use it. Your mother can work with different implementations of remote controlls.

A remote controll works by sending flashes of light to your television set, like sending morse code with a flash light. Only we can't see the light because it is out of our visual range.

Lets define what a remote controll is. A remote control has a button and when I press it the television goes off. Apparently, objects in the real world are defined by what they do, rather then how they do it. Some additional examples of this way of defining this are: An oven heats food. A car starts, moves forward, turns and brakes. A Ideally, one object offers one defined functionalty to its user.

Something is defined by what it does, rather then how it does it.

Besides being able to to turn on or off a television, another quality of a remote controll is that is has a battery. And that battery may be empty or not.
